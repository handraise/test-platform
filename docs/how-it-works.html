AI E2E TESTING

iBud
tests in English, run by AI.

You write what a test should do in plain English. An AI agent drives a real browser to do it, then caches the exact steps so future runs are instant — and re-engages only when something breaks.

THE IDEA

Three problems, one loop

✍️ Authoring is slow

Hand-written selectors and waits rot fast. iBud tests are numbered English steps in a markdown file.

🐢 AI runs are costly

Letting an LLM drive every run burns tokens and time. iBud does it once, then replays deterministically.

💔 Tests break on drift

UIs change and scripts go stale. iBud heals the broken test with AI and re-records it automatically.

THE TEST FILE

A test is plain English

tests/login.test.md
---
name: User can log in
url: /auth/login
tags: [auth, smoke]
---

1. Fill the Email field with "dev@…"
2. Fill the Password field with "…"
3. Click the "Sign In" button
4. Expect the URL to contain "discovery"
5. Expect "Narrative Clusters" visible
  • YAML frontmatter sets the name, start URL, and tags.
  • Numbered steps are instructions — actions and expectations, no selectors.
  • A content hash of the file ties it to its cached script; edit the file and the cache auto-invalidates.
  • Drop a new *.test.md in tests/ and it's discovered automatically.
EXECUTION MODEL

Replay first · heal on drift

RUN 1

AI

Claude reads the steps, looks at the page via the accessibility tree, and drives the browser. Every successful command is recorded.

→
RUN 2…N

Replay

The recorded commands replay directly — no AI, no tokens. Seconds, not minutes.

→
ON FAILURE

Healed

If replay breaks (selector changed, flow moved), the AI re-runs the test, fixes it, and overwrites the cached script.

The slow, expensive AI pass happens rarely — only on the first run or when the app genuinely drifts.

UNDER THE HOOD

What the AI actually controls

🛠️ A small tool surface

The agent gets a handful of tools — snapshot, act, find_act, wait, assert — each a thin call into the agent-browser CLI driving Chrome.

👁️ Accessibility tree, not pixels

snapshot returns roles, names, and refs — so the agent reasons about "the Sign In button", and records stable selectors for replay.

🗂️ Recorded as JSON, not code

Each cached script is a list of agent-browser command arrays in data/scripts/<id>.json — exactly reproducible, trivial to re-record.

🔒 Isolated per run

Every run gets its own browser session, so runs never collide and can go in parallel.

LIVE VISIBILITY

Watch it happen

  • The runner streams events over SSE — step started, passed, failed, screenshot, finished.
  • The dashboard renders a live step timeline with a screenshot captured at each step.
  • A "watch live" toggle embeds agent-browser's own viewport so you see the real browser in real time.
  • Every run is stored in SQLite with its mode, status, duration, and token cost.
